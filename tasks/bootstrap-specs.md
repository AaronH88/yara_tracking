# Task: Split task doc into individual spec files

## What to read

Read the full task document at: `tasks/babytracker-tasks.md`
Read the architecture document at: `tasks/ARCHITECTURE_REF.md` (if it exists — skip if not)

## What to produce

### 1. Individual spec files

For every task in the task document, create a file at:
`tasks/specs/task-{id}.md`

Where `{id}` is the task number, e.g. `0.1`, `1.1`, `1.2` etc.

Each spec file must contain exactly:

# Task {id} — {title}

## Phase
{phase number}

## Description
{full task description from the task doc, verbatim}

## Acceptance Criteria
{the acceptance criteria section verbatim — if stated as "Acceptance criteria:" copy it exactly}

## Verify Scope
{one of: backend | frontend | both}


Rules for `verify_scope`:
- Tasks in Phase 1 (backend only) → `backend`
- Tasks in Phase 2–4 (frontend or mixed) → `frontend` for pure UI tasks, `both` for tasks
  that touch API + UI together
- Phase 6 (PWA/polish) → `frontend`
- Phase 7 (deployment) → `backend`
- When in doubt → `both`

### 2. task-state.json

Create `tasks/task-state.json` with the following structure, populated with
one entry per task in the correct order:

{
  "schema_version": 1,
  "project": "babytracker",
  "created_at": "{current ISO timestamp}",
  "last_updated": "{current ISO timestamp}",
  "final_judge_status": "pending",
  "tasks": [
    {
      "id": "0.1",
      "title": "Initialise Project Structure",
      "phase": 0,
      "state": "PENDING",
      "current_iteration": 1,
      "max_iterations": 3,
      "verify_scope": "both",
      "history": []
    }
  ]
}


Populate one entry per task. Use the exact task IDs and titles from the task doc.
Set all states to `"PENDING"`, `current_iteration` to `1`, `history` to `[]`.
Set `verify_scope` using the same rules as above.

### 3. TASKS.md

Create `tasks/TASKS.md` with a human-readable summary:

# Baby Tracker — Task Status

_Generated by bootstrap — not yet started_

## Phase 0

- ⏳ **0.1** — Initialise Project Structure `PENDING` _iter 1/3_

## Phase 1
...etc


One line per task, all showing `⏳ PENDING`.

## Validation

After writing all files:
1. Count the spec files created and confirm the number matches the task count in the task doc
2. Confirm `task-state.json` is valid JSON (`python3 -c "import json; json.load(open('tasks/task-state.json'))"`)
3. Print a summary: list every spec file created, one per line
